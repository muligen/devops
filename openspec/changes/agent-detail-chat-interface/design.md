## Context

当前 AgentTeams 系统的 Agent 详情页采用传统列表方式展示最近任务，用户执行任务需要跳转多个页面或打开模态框。这种交互方式对于需要频繁执行命令的运维人员来说效率较低。系统已具备以下能力：

- WebSocket 实时推送支持（`realtime-push` spec）
- 任务执行功能完整（`task-execution` spec）
- 快速任务执行对话框存在（`quick-task-exec` spec）

目标是在不改变后端核心逻辑的前提下，通过前端重构提供更直观的聊天式命令交互体验。

## Goals / Non-Goals

**Goals:**
- 在 Agent 详情页提供聊天式命令终端界面
- 支持实时命令输入和结果展示
- 整合任务执行功能到单一页面
- 提升运维人员的操作效率和体验
- 复用现有后端 API 和 WebSocket 能力

**Non-Goals:**
- 不修改后端任务执行核心逻辑
- 不改变用户认证和授权机制
- 不影响现有任务管理页面功能（该页面保留用于批量操作和历史查询）
- 不引入新的前端框架或重型依赖

## Decisions

### 1. 聊天界面布局设计

**决策：采用左右分栏布局，左侧显示历史记录，右侧固定命令输入框**

**理由：**
- 左侧历史记录区域可以滚动查看大量命令和结果，右侧输入框固定位置方便持续操作
- 符合常见终端模拟器的布局习惯，用户学习成本低
- 响应式设计支持移动端侧边栏切换

**替代方案考虑：**
- 上下布局：命令记录在上，输入框在下 - 被拒绝，因为命令输出通常较长，会挤压可用空间

### 2. 消息气泡样式设计

**决策：用户消息在右，机器消息在左，使用不同颜色区分**

**理由：**
- 符合即时通讯应用的通用交互模式
- 右侧蓝色/紫色表示用户，左侧灰色/绿色表示系统/机器，视觉层次清晰
- 支持深色主题下的对比度优化

### 3. 实时输出实现方式

**决策：使用现有 WebSocket 机制，在任务执行期间推送 `output_chunk` 消息**

**理由：**
- 后端已有 Agent 输出流式推送能力（`task-execution` spec 定义）
- 前端通过 WebSocket 接收并追加到消息气泡
- 无需新增 API，保持架构简洁

### 4. 命令历史存储

**决策：使用 localStorage 存储命令历史，每个 Agent 独立历史**

**理由：**
- localStorage 容量足够（5MB），可存储数千条命令
- 不会占用服务器资源
- 支持离线查看历史命令（即使 Agent 离线也可以查看之前的历史）
- 按 Agent ID 分隔存储，避免不同 Agent 的命令混淆

### 5. 组件架构

**决策：拆分为 React 组件树**

```
AgentTerminal
├── TerminalHeader (终端顶部：Agent 信息、状态指示)
├── MessageList (消息列表)
│   ├── CommandMessage (命令消息 - 用户)
│   ├── ResultMessage (结果消息 - 机器)
│   │   ├── OutputContent (输出内容 - 支持流式更新)
│   │   └── StatusIndicator (状态指示)
│   └── ErrorMessage (错误消息)
├── QuickCommands (快捷命令栏)
│   └── CommandButton (单个命令按钮)
└── CommandInput (命令输入区域)
    ├── CommandLine (命令行)
    └── ExecuteButton (执行按钮)
```

**理由：**
- 组件拆分清晰，便于维护和测试
- 支持复用（如 CommandMessage 可在其他场景使用）
- 符合 React 单向数据流原则

### 6. 状态管理

**决策：使用 React Query 处理后台任务状态，Zustand 管理本地消息状态**

**理由：**
- React Query 已用于现有代码中的 API 请求和缓存
- Zustand 轻量级，适合管理 UI 本地状态
- 不引入过多的状态管理复杂度

## Risks / Trade-offs

### 风险 1: 大量输出导致的性能问题

**风险描述：** 当命令输出特别大时（如 `ls -R` 大目录），可能导致页面卡顿

**缓解措施：**
- 限制单条消息的显示长度，超过阈值提供"展开更多"功能
- 使用虚拟滚动（react-window）优化长列表渲染
- 支持输出内容分页

### 风险 2: WebSocket 断连导致输出中断

**风险描述：** 网络波动时 WebSocket 可能断连，导致输出流中断

**缓解措施：**
- 利用现有 WebSocket 重连机制（`realtime-push` spec）
- 断连后自动通过轮询 API 获取最新任务状态
- 显示断连状态，支持手动刷新

### 风险 3: 跨 Agent 命令混淆

**风险描述：** 用户在多个 Agent 详情页之间切换，可能在不同 Agent 中执行了错误的命令

**缓解措施：**
- 终端顶部始终显示当前 Agent 名称和 ID
- 命令历史按 Agent ID 独立存储
- 在命令确认时高亮显示目标 Agent

## Migration Plan

**部署步骤：**

1. **前端部署（零停机）：**
   - 新增组件和样式，不影响现有页面
   - 在 Agent 详情页添加新的"终端"标签页选项
   - 保留原有"最近任务"列表，逐步引导用户转向新界面

2. **灰度发布：**
   - 先对部分用户启用新的终端界面
   - 收集用户反馈，优化交互细节
   - 全量发布后，可将最近任务列表移至二级页面

3. **回滚策略：**
   - 通过路由参数控制界面模式
   - 如遇严重问题，可快速切换回原有列表视图

**数据迁移：**
- 无需数据库迁移，纯前端改造
- 历史任务数据可继续在任务管理页面查看

## Open Questions

1. **是否需要在 Agent 详情页保留"最近任务"列表视图？**
   - 决议：保留作为备用视图，在新终端界面稳定后考虑移除

2. **是否支持命令自动补全？**
   - 决议：基于用户历史命令提供基本补全，不支持系统级补全（复杂度高）

3. **是否支持多 Agent 并发命令执行？**
   - 决议：当前不支持，单 Agent 界面设计为专注操作一个 Agent。批量操作仍保留在任务管理页面
